📈 我哋點樣用 AI + 大數據分析六合彩？
2026-05-12

作為一個對數據科學同彩票有興趣嘅開發者，我發現市面上嘅六合彩分析工具大多數只係簡單嘅「隨機號碼產生器」，完全冇考慮歷史數據嘅統計規律。於是乎，我決定自己動手，整一個 真正基於大數據 + 機器學習概念 嘅分析系統。

呢個系統唔會保證你中獎（冇任何人可以），但佢會俾你一個科學化嘅參考——邊啲號碼最近熱、邊啲係盲門、邊種組合歷史上最常見。你可以自己決定跟唔跟。

系統三大核心：

  • 自動化數據收集：每日 22:03 GitHub Actions 定時抓取公開數據，寫入 Google Sheets。
  • 完整統計分析：熱冷號、盲門、顏色分佈、奇偶大小和值區間尾數，全部圖表化。
  • AI 預測引擎：加權評分（熱號/冷號/避免重複）+ 組合過濾（和值/奇偶）。

真實命中率（初步）：
單式 73 注暫未中 3 個以上；複式 20 碼平均包 1.33 個中獎號；膽拖拖碼平均 1.00 個。
會繼續收集數據微調演算法。

技術改進：數據源更換、NaN 修復、Google Sheets API 踩坑、定時任務重啟等（詳見下方日誌）。

🎯 真實命中率分析上線
2026-05-12

加咗 stats_real_accuracy.html 呢頁，夾埋 Google Form 收集嘅用戶預測數據，同真實開獎結果對比，計返單式、複式、膽拖嘅實際命中分佈。

  • 單式：暫時累積 73 注，命中 3 個或以上號碼嘅比例係 0%（仲要更多數據先睇得真）。
  • 複式：推薦 20 個號碼平均包含 1.33 個中獎號碼。
  • 膽拖：拖碼平均命中 1.00 個。

之後會根據呢啲真實回饋，調整預測算法嘅參數，希望慢慢提升中獎率。

📊 統計頁大升級：盲門、顏色分佈、冷熱號
2026-05-11

大幅強化 stats.html 嘅分析功能：

  • 盲門：定義做「最近 180 期從未出現過嘅攪出號碼」，用灰色波波顯示。
  • 顏色分佈：跟返官方號碼顏色規則（紅/藍/綠），畫咗個圓餅圖。
  • 熱冷號定義：熱號 = 最近 20 期出現次數最多嘅前 10 個號碼；冷號 = 出現最少嘅前 10 個。
  • 另外仲加咗和值直方圖、區間分佈、尾數分佈等圖表。

所有圖表都係每次 refresh 頁面時直接讀 Google Sheets 嘅最新數據，唔使人手更新。

🔄 換數據源＋重寫解析器
2026-05-11

原本靠一個公開數據網頁，點知佢改咗版面結構，搞到解析唔到，期號同號碼出晒 NaN。解決方法：

  • 改用另一個結構穩定嘅公開數據頁。
  • 重新寫 Python 爬蟲，用正則表達式直接由 <tbody> 裡面抽出第一個有效 <tr>。
  • 加咗幾層備用解析規則，萬一第時條 markup 輕微改動都仲頂得住。

而家每日自動（香港時間 22:03）執行，成功將最新一期寫入 Google Sheets。

📁 數據儲存嘅黑暗歷史：Google Sheets API 同 Service Account 嘅搏鬥
2026-05-11
  • Service Account 權限：要 share 俾 xxx@iam.gserviceaccount.com，唔係自己 email。
  • GitHub Secrets 格式：JSON 金鑰要壓成一行先唔會爛。
  • gspread 版本問題：指定 gspread==5.12.0。
  • Apps Script 中伏：timeout 同 debug 困難，最後放棄，直接用 Google Sheets API v4。
  • 期號含有 '/' 會被轉成日期：寫入前加單引號 '26/049 解決。

而家流程穩定：Python + gspread + Service Account，每日 insert_row 到第二行。

🐛 修正首頁期號同 NaN 問題
2026-05-10

之前有用戶反映主頁顯示「26/047 NaN,9,17,...」，查完發現：

  • Google Sheets 其實已經有期號欄位，但前端 loadCsvData() 冇讀嗰一欄，反而用 47 - index 嚟自己計期號。
  • 修正方法：擴大讀取範圍到 A:H 列，正確解析期號同 6+1 個號碼。
  • 同時改埋 renderSingle 入面最新一期同最近 5 期嘅顯示方式。

而家主頁正確顯示「26/049 9,17,26,41,42,47 + 特別號 8」。

2026-05-09

  • Google Sheets API：前端直接讀取試算表。

🧠 預測算法核心：加權評分 + 組合過濾
2026-05-08
  • 加權評分：近期權重高（指數衰減），考慮熱號、冷號、避免重覆上期。
  • 單式：由最高分嘅 20 個號碼隨機抽 6 個（固定種子可重現）。
  • 複式：直接拎最高分 N 個號碼。
  • 膽拖：先揀膽碼再揀拖碼。
🗣️ 由「齋 prompt」到「AI 寫 code，我負責設計」
2026-04-15

🤖 第一階段：只係同 Copilot 對話
最初我冇諗過整網站，只係對住 Copilot 落 prompt，叫佢：「根據歷史數據，先分析過去十年嘅號碼分佈；然後再加入近期 15–20 期嘅數據做交叉比對；最後用防守方法、江恩週期、移動平均、馬可夫鏈，搵出最大機會率嘅號碼」。嗰陣時我只係當一個數學實驗，所有「模型」都係 Copilot 幫我生成嘅文字描述同虛擬碼，我一行真正嘅程式碼都冇寫過。

💡 第二階段：決定整一個真嘅系統，我做設計者
覺得個概念幾有趣，的起心肝要將佢變成一個任何人都用到嘅網站。但自己唔想由零寫 code，於是轉換角色：我係 **設計者**，負責規劃功能、流程、視覺方向，然後交俾 AI（例如 ChatGPT、Copilot）幫我生成所有 HTML、CSS、JavaScript。我只係列出需求、決定版面點排、選擇用邊種圖表、定義咩係「熱號」「冷號」「盲門」，然後將 AI 產生嘅 code copy & paste 上 Netlify 試行。

🐞 第三階段：debug 嘅關鍵——畀方向 AI
當 AI 寫嘅 code 出錯，我發現佢成日會陷入一個重複 loop：同一個問題改完又錯、錯完又改，來回好多次都解決唔到。呢個時候，我嘅角色就變得好重要：我要仔細睇 error message、推測可能嘅原因，然後俾一個具體嘅 debug 方向畀 AI（例如「你試下檢查 variable scope」、「應該係 async/await 嘅問題，試下重組 Promise」）。AI 收到明確方向之後，通常一兩次就搞掂。如果淨係掉句「唔 work，幫我改」，佢就會無限 loop。所以 debug 階段，設計者嘅邏輯思維同埋俾方向嘅能力，比起純粹盲試更關鍵。

🔧 第四階段：加入真實數據同自動化，繼續做設計者
網站要有真實數據先有意義。我設計咗數據流程：每日定時抓取公開結果 → 存入 Google Sheets → 前端讀取顯示。我叫 AI 幫我寫 Python 爬蟲，用 GitHub Actions 定時執行，並透過 Google Sheets API 儲存。前端點樣讀取試算表、點樣顯示預測同統計，都係我講個概念，然後 AI 生成 JavaScript。debug 嗰陣又係由我俾方向，先突破到 AI 嘅 loop。

📈 第五階段：統計、SEO、開發者日誌，全部由我設計、AI 實作
系統成形之後，我設計咗統計頁面要顯示嘅指標（熱冷號、盲門、顏色分佈、和值直方圖），又叫 AI 加真實命中率追蹤功能、開發者日誌，仲有基本 SEO 設定（sitemap、robots.txt、meta tags）。每一個功能都係我先畫出「要咩效果」，然後 AI 生 code，我負責整合同 deploy。遇到 AI 改極都改唔好嘅問題，我就會自己分析，畀個方向佢，通常就好快解決。

🎯 回顧
由純粹同 Copilot 傾計，到最後 deploy 出一個完整嘅網站，我嘅角色係 **設計者**，同時係 **debug 方向嘅畀出者**。我自己冇寫過一行核心邏輯，但需要具備足夠嘅邏輯同除錯 sense，先可以帶領 AI 走出無限 loop。呢個開發日誌，係一份「普通人以設計者身份，用 AI 由零到一建立網站」嘅真實記錄。
